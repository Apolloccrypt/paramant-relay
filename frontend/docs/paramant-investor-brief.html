The Solution — Ghost Pipe Protocol

Different architecture Not a better Zivver

Not a better version of Zivver. A different architectural answer.

Zivver, Tresorit, and WeTransfer are all built on the same assumption: a server stores your files, and security is achieved by encrypting access to those files. The relay is a database. Ghost Pipe eliminates the database. The relay is a conduit — it holds only ciphertext, only in RAM, only until the recipient downloads it. Then it's gone.

The Ghost Pipe Protocol

How it works

  • Files encrypted client-side with ML-KEM-768 (NIST FIPS 203)
  • Relay receives and stores only ciphertext
  • Ciphertext held in RAM only — never written to disk
  • Destroyed after one read (burn-on-read)
  • Merkle CT log proves delivery without storing content
  • ML-DSA-65 post-quantum signature on every proof

What this means in practice

  • Nothing to breach — relay holds no persistent content
  • Nothing to subpoena — no content to compel production of
  • Nothing to harvest — HNDL attack eliminated by design
  • Verifiable — public CT log, cryptographic proof per transfer
  • Sovereign — EU infrastructure, no US CLOUD Act

Competitive Moat

  • PQ Post-quantum by default. Competitors have "roadmaps". ML-KEM-768 is live in production today. The migration window is open — most organisations have not started.
  • 0S Zero storage architecture. This is not achievable by Zivver via a software update. Their entire product is built on stored files. Eliminating storage requires a different product from the ground up.
  • CT Public Certificate Transparency log. No competitor has this for file transport. Every transfer produces a cryptographic inclusion proof, verifiable by any party without accessing content.
  • OS Open source (BUSL-1.1). Regulated industries require auditability. Source-available builds trust that no closed-source product can match. Prevents competitor fork-and-resell.
  • SH Self-hostable. Hospitals and factories can run their own relay on their own infrastructure. No vendor dependency. Even if Paramant as a service shuts down, every self-hosted deployment keeps running.
