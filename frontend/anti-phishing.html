USE CASE — SECURITY

Breaking the mass-phishing pattern.

Mass phishing campaigns rely on one link reaching thousands of victims. Burn-on-read makes that pattern structurally impossible. The first recipient opens the file. Everyone else sees HTTP 410.

01 — the attack pattern

Trusted-domain phishing.

Attackers send mail with links to legitimate fileshare platforms. SharePoint. OneDrive. WeTransfer. Dropbox. DocuSign. The domain is real. The file behind it is not.

The victim sees a domain they trust. Clicks. Is asked to sign in, or downloads what looks like a document. Credentials are stolen, or malware is delivered, or OAuth permissions are granted to a malicious app.

This is the pattern that killed Firefox Send in 2020. Legitimate infrastructure, abused for mass distribution. Mozilla took the service offline, then layoffs removed the team that could have fixed it. The pattern still works today, just on other platforms.

02 — what paramant breaks

Mass distribution, structurally.

An attacker shares a single Paramant link with 10,000 targets. The first target opens it. The file is read from RAM, served once, and deleted. The remaining 9,999 targets see HTTP 410 Gone.

There is no retry. There is no re-download. There is no caching layer that preserved the file. The file existed in one relay’s RAM for a few seconds and is now mathematically unreachable.

Mass phishing campaigns structurally cannot work on Paramant. The economics of the attack collapse at scale.

03 — honest limitations

What Paramant does not solve.

Paramant is delivery infrastructure, not endpoint security.

If a recipient trusts a sender who turns out to be malicious, the file still arrives. If the sender’s device is compromised and the attacker uses their API key, the file is signed as legitimate. If the recipient’s machine has no antivirus, a malicious payload still executes locally.

Paramant solves the transport layer. Endpoint protection stays with the recipient’s tooling. We make one class of attack impossible. We do not claim to make all classes impossible.

Anyone who claims their product prevents phishing entirely is selling you something.

04 — three scenarios

How the attacks fail.

A

Credential phishing via fake login UI

Attacker sends a mail pretending to be from a fileshare service, linking to a spoofed login page. Victim enters credentials on the fake page. Attacker now has access to the victim’s real account.

On Paramant: this attack has no surface. Paramant has no password. No login form. No “Sign in with Paramant” button. There is nothing for the attacker to spoof.
B

Malware mass distribution

Attacker uploads a malicious payload disguised as an invoice, shares one link with 10,000 targets over two weeks. Victims trust the domain, download, execute.

On Paramant: the first victim triggers the burn. The file is delivered once and destroyed. Victims 2 through 10,000 see HTTP 410. The attacker has to upload a new file for every single target, which defeats the economics of mass campaigns.
C

Sender spoofing

Attacker poses as a trusted colleague or supplier, sends a file expecting the victim to open it without verification.

On Paramant: every transfer is signed with ML-DSA-65 and logged in a public Merkle tree. The recipient can cryptographically verify the file came from the claimed sender. The verification is not automatic — it requires the recipient to check. But the proof exists and cannot be forged.
05 — comparison

How platforms handle mass phishing abuse.

WeTransfer Yes — 7-day retention by default Yes — email-based auth No
